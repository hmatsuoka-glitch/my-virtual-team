# Bug Fix - STEP 2: Analyze（原因解析）

**担当**: 該当する実装担当エージェント（riku/ao/kuu）
**前段**: STEP 1 報告
**後段**: STEP 3 修正

---

## 🎯 目的

**症状ではなく根本原因（Root Cause）を特定する。**
場当たり的な修正は再発を招く。

---

## 📋 解析フロー

### STEP 2-1: 再現確認

```
- [ ] 報告書通りの手順で再現できる
- [ ] 再現率（毎回？/時々？/特定条件のみ？）
- [ ] 再現環境（dev/staging/prod のどこで起きるか）
```

### STEP 2-2: 5 Whys（5回なぜを問う）

```
症状: ログインができない

なぜ? → 401エラーが返る
なぜ? → トークン検証が失敗する
なぜ? → トークンの署名が不正
なぜ? → 環境変数 JWT_SECRET が読まれていない
なぜ? → 最新リリースで .env のキー名が変更されたが
       Vercel 環境変数の更新が漏れた

根本原因: デプロイ手順の不備（環境変数の同期チェックなし）
```

### STEP 2-3: 影響範囲の特定

```
- [ ] このバグでデータが壊れた可能性は？
- [ ] 他のユーザーにも発生している？
- [ ] 過去のリリースのどこから入った？（git bisect）
- [ ] 他の機能への波及は？
```

### STEP 2-4: 修正方針の決定

| 修正タイプ | 説明 | 適用条件 |
|-----------|------|---------|
| **Hotfix** | 即座に当てる応急処置 | P0で即対応必要 |
| **Proper Fix** | 根本原因への対応 | P1以下で時間ある |
| **Workaround + Fix** | 応急 → 後で根本 | P0だが根本修正に時間かかる |

---

## 📋 解析レポート出力

```markdown
# Bug #XXX 解析レポート

## 再現確認
- 再現率: 100% / 50% / 0%
- 再現環境: production
- 再現手順: 報告書通り

## 5 Whys
1. なぜ? → ...
2. なぜ? → ...
...
**根本原因**: ...

## 影響範囲
- 影響ユーザー: 全ログインユーザー
- データ破損: なし
- 波及機能: 認証必須機能全て
- バグ混入リリース: v1.2.3 (2026-XX-XX)

## 修正方針
タイプ: Hotfix + Proper Fix
- Hotfix: 環境変数を Vercel に追加（10分）
- Proper Fix: デプロイチェックリストに環境変数同期を追加（1時間）

## テスト計画
- [ ] 環境変数なしで起動 → 起動時にエラーで止まるテストを追加
- [ ] CI で .env.example と prod 環境変数の差分検出
```

- [ ] 影響範囲が明確
- [ ] 修正方針（Hotfix / Proper Fix）が決定
- [ ] 再発防止のテスト計画がある
