# Bug Fix - STEP 4: Verify（検証）

**担当**: Mio（QA）+ Sora（COO）
**前段**: STEP 3 修正
**後段**: ユーザー納品

---

## 🎯 目的

修正が **本当に直っている** ことを確認 + **再発しない** ことを保証する。

---

## 📋 検証チェックリスト

### 1. バグの解消確認（Mio）

- [ ] 元の再現手順でバグが起きないことを確認
- [ ] 自動テストが緑（CI / ローカル両方）
- [ ] 修正前と修正後のスクリーンショットを比較
- [ ] 修正コードが Code Review を通過

### 2. リグレッション確認（Mio）

- [ ] 既存の全テストスイートを実行
- [ ] 主要ユーザーフローを E2E テスト
- [ ] 修正に関連する隣接機能の動作確認

```
例: 認証バグの修正なら...
- [ ] ログイン
- [ ] ログアウト
- [ ] パスワードリセット
- [ ] セッション期限切れ
- [ ] 認証必須ページへのアクセス
```

### 3. 本番デプロイ前チェック（Kuu）

- [ ] staging環境で再現 → 修正確認
- [ ] パフォーマンスへの影響なし（レスポンス時間等）
- [ ] エラーログに新しい異常がない
- [ ] ロールバック手順を準備

### 4. 再発防止策（全員）

- [ ] テストカバレッジが追加されている
- [ ] 同種のバグを検出する CI チェックがある
- [ ] チェックリスト/ドキュメントに反省点を反映
- [ ] チームに共有（postmortem）

### 5. ユーザー視点の検証（Sora）

- [ ] ユーザーが報告した症状と一致した修正か
- [ ] ユーザー体験を損なう副作用がないか
- [ ] 修正によって新たな脆弱性が生まれていないか

---

## 📋 Postmortem（事後分析）レポート

P0/P1 バグの場合は必須で作成：

```markdown
# Bug #XXX Postmortem

## 発生
- 報告日時:
- 発覚から修正までの時間:
- 影響を受けたユーザー数:
- ダウンタイム:

## タイムライン
- 10:00 ユーザー報告
- 10:05 Kai が再現確認
- 10:10 Ao が解析開始
- 10:30 根本原因特定
- 10:45 Hotfix デプロイ
- 11:00 Proper Fix PR 作成

## 根本原因
（5 Whys の結果）

## 何がうまくいったか
- 例: 監視アラートで即検知できた

## 何がうまくいかなかったか
- 例: 環境変数の同期チェックがなかった

## アクションアイテム
- [ ] CI に環境変数同期チェック追加（担当: kuu, 期限: 来週）
- [ ] デプロイチェックリスト更新（担当: kai, 期限: 明日）
- [ ] 同種バグの自動検出ルール追加（担当: mio, 期限: 来週）
```

---

## ✅ 検証完了の判定基準

- [ ] 元のバグが再現しない
- [ ] リグレッションなし
- [ ] 再発防止策が実装済み
- [ ] Postmortem 作成（P0/P1のみ）
- [ ] Sora の最終OK

→ ユーザーへ納品（修正PRマージ + リリース）
