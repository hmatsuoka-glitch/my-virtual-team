# STEP 3: タスク分解（Tasks）

**担当**: Kai（PM）
**前段**: STEP 2 設計
**後段**: STEP 4 実装

---

## 🎯 目的

設計書を**実行可能なタスク粒度**に分解。各タスクは独立してテスト可能。

---

## 📋 タスク分解の原則

### 原則1: INVEST 原則
各タスクが以下を満たすこと：

- **I**ndependent: 独立している（順番が交換できる、または依存関係明示）
- **N**egotiable: 詳細は実装時に調整可能
- **V**aluable: ユーザー価値がある（技術的でなくビジネス的）
- **E**stimable: 工数見積もり可能
- **S**mall: 1〜2日以内で完了
- **T**estable: テスト基準が明確

### 原則2: 並列実行可能性を最大化

タスクを **依存グラフ** に整理：

```
[タスクA: DB schema]
         ↓
    ┌────┴────┐
    ↓         ↓
[B: User API] [C: Auth基盤]
    ↓         ↓
    └────┬────┘
         ↓
    [D: ログインUI]
```

**Aは順次必須、B/Cは並列可能、Dは両方完了後**

---

## 📋 タスクの記述フォーマット

各タスクをチケット形式で記述：

```markdown
### Task #001: ユーザーテーブルのDB Schema作成

**担当**: ao
**優先度**: P0（ブロッカー）
**見積**: 2時間
**依存**: なし
**並列可能タスク**: #002, #003

**Done条件**:
- [ ] users テーブルが Supabase にマイグレーションされる
- [ ] RLS ポリシーが適用される
- [ ] テストデータが seeds/ から投入できる
- [ ] migration ファイルが Git管理されている

**実装手順** (TDD):
1. tests/db/users.test.ts でテーブル存在テスト
2. supabase/migrations/001_users.sql 作成
3. supabase db push
4. テスト通過確認

**受け入れテスト**:
- `SELECT * FROM users LIMIT 0;` が成功
- 不正なemail挿入時にCHECK制約エラー
- 認証されていないユーザーがSELECTできない（RLS）
```

---

## 📊 タスク優先度

| 記号 | 意味 | 例 |
|------|------|-----|
| **P0** | ブロッカー（これが終わらないと他が進まない） | DB schema |
| **P1** | 主要機能（リリースに必須） | 認証・コア機能 |
| **P2** | 重要だがリリース後でも可 | パフォーマンス改善 |
| **P3** | あると良い（Nice to have） | アニメーション |

---

## ✅ タスク分解完了の判定基準

- [ ] 全タスクに担当エージェントが決まっている
- [ ] 各タスクの Done 条件が具体的
- [ ] 依存関係が明示されている
- [ ] 並列実行可能なタスク群が識別されている
- [ ] 優先度（P0〜P3）が付いている
- [ ] 受け入れテストが書かれている

---

## 🚨 アンチパターン

❌ 「フロントエンド実装」のような大きすぎるタスク
❌ Done条件なし（「完了」の定義が人によって違う）
❌ 依存関係なし（実装で詰まる）
❌ テスト基準なし（後でテストが書けない）
