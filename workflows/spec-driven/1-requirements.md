# STEP 1: 要件定義（Requirements）

**担当**: Nao（システム開発部・要件定義/アーキテクト）
**前段**: Kai（要件整理レポート）
**後段**: STEP 2 設計

---

## 🎯 目的

ユーザーが本当に欲しいものを正確に把握し、後工程で迷わない要件書を作る。
**「思った通りのシステムを作る」精度の80%はここで決まる。**

---

## 📋 要件定義書の必須セクション

### 1. プロジェクト概要
- **目的**: なぜこのシステムを作るのか（ビジネスゴール）
- **対象ユーザー**: 誰が使うのか（ペルソナ・利用シーン）
- **成功基準**: 何ができたら成功か（KPI・受け入れ基準）

### 2. 機能要件（What）

各機能に対して以下を定義：

```
【機能名】〇〇機能
【概要】1〜2行で機能を説明
【ユーザーストーリー】
  〇〇として
  △△したい
  なぜなら□□だから
【受け入れ基準】（Given-When-Then 形式）
  Given: 前提条件
  When: ユーザーがこう操作する
  Then: こう振る舞う
【優先度】MUST / SHOULD / COULD / WON'T (MoSCoW)
```

### 3. 非機能要件（How well）

| 項目 | 内容 | 数値目標 |
|------|------|---------|
| パフォーマンス | レスポンス速度 | 例: API < 500ms (p95) |
| スケーラビリティ | 同時接続数 | 例: 1,000 ユーザー |
| 可用性 | アップタイム | 例: 99.5% |
| セキュリティ | 認証・認可 | 例: OAuth 2.0 + RBAC |
| アクセシビリティ | 対応レベル | 例: WCAG 2.1 AA |
| ブラウザ対応 | 対象 | 例: Chrome/Safari/Edge 直近2バージョン |
| データ保持 | 期間・形式 | 例: 1年・CSV出力可 |

### 4. 制約事項
- **技術制約**: 必須技術、禁止技術
- **予算制約**: ホスティング費用上限
- **時間制約**: 納期
- **法的制約**: 個人情報保護法等

### 5. スコープ外（重要）
**「やらないこと」を明示する**。
スコープ外を書かないと「これもできるよね？」が積み重なって破綻する。

```
スコープ外:
- ❌ モバイルネイティブアプリ（Webのみ）
- ❌ 多言語対応（日本語のみ）
- ❌ 決済機能（別フェーズ）
```

### 6. リスク分析

| リスク | 影響 | 確率 | 対策 |
|--------|------|------|------|
| 例: API仕様変更 | 高 | 中 | 抽象化レイヤーで吸収 |

---

## ✅ 要件定義完了の判定基準

以下を全て満たして次STEPへ：

- [ ] ユーザーストーリーが全機能分書かれている
- [ ] 受け入れ基準（Given-When-Then）が具体的
- [ ] 非機能要件に**数値目標**がある
- [ ] スコープ外が明記されている
- [ ] **ユーザーが要件書を読んでOKと言った**
- [ ] 曖昧な単語（「使いやすい」「速い」等）に数値定義がある

---

## 🚨 アンチパターン（やってはいけないこと）

❌ 「ログイン機能」とだけ書く（→ どんなログイン？認証方法は？）
❌ 「使いやすいUI」（→ 使いやすさを数値化できない）
❌ スコープ外を書かない（→ 後で揉める）
❌ 受け入れ基準なしで進める（→ テスト書けない）
❌ 非機能要件を後回し（→ 完成後に「遅い！」が発覚）
