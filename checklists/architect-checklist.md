# Architect Checklist（設計完了チェック）

**担当**: Nao（アーキテクト）
**使用タイミング**: STEP 2 設計完了時

設計書がこのチェックリストを全て満たすまで STEP 3（タスク分解）に進んではいけない。

---

## 🏗️ システム構成

- [ ] アーキテクチャ図がある（コンポーネント間の関係が分かる）
- [ ] 技術スタックが明記されている（言語・FW・DB・インフラ）
- [ ] 各技術選定の理由が書かれている
- [ ] 外部サービス依存が一覧化されている（API・SaaS）

## 📁 ディレクトリ構成

- [ ] フォルダ階層が明示されている
- [ ] 各フォルダの責務が一文で説明されている
- [ ] 命名規則（ファイル名・関数名・変数名）が定義されている

## 🗄️ データベース設計

- [ ] ER図がある
- [ ] 全テーブルにDDL（CREATE文）がある
- [ ] 主キー・外部キーが全テーブルに定義
- [ ] インデックス戦略が記載されている
- [ ] CHECK制約・UNIQUE制約・NOT NULL制約が明示
- [ ] RLS（Row Level Security）ポリシーが定義されている
- [ ] マイグレーション方針が記載されている

## 🔌 API設計

- [ ] 全エンドポイントが一覧化されている
- [ ] 各エンドポイントに以下が定義：
  - [ ] HTTP メソッド・パス
  - [ ] リクエストスキーマ（Zod推奨）
  - [ ] レスポンススキーマ（成功・全エラー）
  - [ ] 認証要件（公開 / 認証必須 / ロール限定）
  - [ ] レート制限
- [ ] エラーコードの一覧と意味
- [ ] APIバージョニング戦略

## 🎨 画面設計

- [ ] 画面遷移図がある
- [ ] 各画面のワイヤーフレーム or 説明
- [ ] レスポンシブ対応方針（PC / Tablet / Mobile）
- [ ] アクセシビリティ対応方針（WCAG レベル）
- [ ] 状態管理（ローディング・エラー・空状態）

## 🔐 セキュリティ設計

- [ ] 認証方式が明確（OAuth / Session / JWT等）
- [ ] 認可方式が明確（RBAC / ABAC / リソースベース）
- [ ] 入力検証ライブラリが指定（Zod等）
- [ ] XSS対策方針
- [ ] CSRF対策方針
- [ ] SQL Injection対策方針
- [ ] シークレット管理（環境変数 / Vault）
- [ ] PII（個人情報）の取り扱い方針
- [ ] 監査ログの設計

## 🧪 テスト戦略

- [ ] テストの種類と目標カバレッジが定義
  - [ ] Unit: 80%+
  - [ ] Integration: 主要API全網羅
  - [ ] E2E: 主要ユーザーフロー
- [ ] テストフレームワーク選定（Vitest / Jest / Playwright等）
- [ ] テストデータ戦略（fixtures / factories）
- [ ] CI/CDでのテスト実行戦略

## 🚀 デプロイ戦略

- [ ] 環境（dev / staging / production）が定義
- [ ] CI/CDパイプライン設計
- [ ] デプロイ手順（自動 vs 手動承認）
- [ ] ロールバック手順
- [ ] 環境変数管理方針

## 📊 監視・ログ

- [ ] エラー監視（Sentry / Vercel Analytics等）
- [ ] パフォーマンス監視
- [ ] ログレベル戦略（debug / info / warn / error）
- [ ] アラート条件

## 🌐 非機能要件への対応

要件書の非機能要件 1 つずつに対して：

- [ ] パフォーマンス目標 → 設計上の対応策
- [ ] スケーラビリティ目標 → 設計上の対応策
- [ ] 可用性目標 → 設計上の対応策
- [ ] アクセシビリティ目標 → 設計上の対応策

## 📝 ドキュメント

- [ ] README.md（セットアップ手順）
- [ ] CONTRIBUTING.md（開発ルール）
- [ ] ADR（Architecture Decision Records）の方針

## 🔍 リスク・前提条件

- [ ] 技術的リスクが洗い出されている
- [ ] 前提条件が明示されている
- [ ] 未確定事項のリストがある

---

## 🚨 不合格になる典型例

❌ ER図なし → テーブル間の関係性が不明
❌ APIのエラーレスポンス未定義 → 実装者がバラバラに作る
❌ テスト戦略なし → 後で「テストどう書くの？」が発生
❌ セキュリティ設計が「あとで考える」 → 重大な脆弱性

---

## ✅ 全て満たしたら

→ STEP 3: タスク分解（`workflows/spec-driven/3-tasks.md`）へ進む
