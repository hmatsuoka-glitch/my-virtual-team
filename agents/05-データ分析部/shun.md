### 2026-05-03
- **「データは正確 ≠ アクション結論が見える」の落し穴**：Airwork応募数、GA4到達率、CVRなど数値は正確に報告しても、クライアント採用担当者から「で、何を変えればいいですか？」という質問が返ってくることが多い。根因：グラフ・表は「事実の提示」に過ぎず、「どの数値が悪い → なぜ悪い → 何をする」というストーリーラインが欠ける。改善：数値毎に「評価（良い/悪い）」「原因仮説」「推奨施策」を1行セットで添記。表の下に「意思決定フロー」（例：応募CVRが低い → LP滞在時間が短い → ファーストビューの訴求見直し）を記載することで、クライアントが「理解 → 判断 → 実行」へ進む。
- **グラフの「見やすさ」と「正確性」の二律背反**：折れ線グラフをY軸0-1000でスケーリングすると、微小な変動が誇大に見える。一方、Y軸を自動スケーリングすると、元のデータの正確な比率が見える。クライアントの意思決定を支援する観点では、「軸の選択理由を明示する」ことが信頼につながる。回避策：グラフの下に「Y軸範囲：0-1000に固定→月間最大応募数を基準に設定」のように、スケーリング判断を明文化。
- **Akari・Ryota への「前置き」の重要性**：Akariにデータテーブルを渡す際、計算式・条件・仮定（「外れ値除外n=3」「サンプル数1000未満は集計外」）を省略すると、Akariが独自判断で数値を修正してレポート品質が劣化。Ryotaに渡す場合も「クライアントには『応募CVRが前月比-8%だが、流入品質の改善余地あり』と説明してください」と前置きしないと、数字だけが一人歩きしてクライアント信頼を損なう。回避策：全成果物に「読み手への前置き・背景」を1段落追加。

### 2026-05-06
- **指標の「分母」曖昧さ：CVRの計算ベースを統一しないと月次比較が無意味に**：「応募CVR = 応募数÷閲覧数」が standard だが、実装では「応募数÷ユニークユーザー数」「応募数÷セッション数」など異なる分母が混在。4月は「ユニークユーザー」ベース、5月は「セッション」ベースでレポートすると、数値が20%以上乖離しても「変化した」と誤判定。回避策：月初に「分析定義書」を更新し、全指標の分母・計測起点を明記。Akariへの引き継ぎ時に「このCVRは〇〇ベース」と前置き。
- **季節性・曜日パターンの見落とし**：GA4の日別推移で「月曜日の応募が多い」という傾向を見つけても、母数や構成比を確認せず「月曜狙いの施策が効く」と結論づけると落し穴。実は全体応募数は増えておらず、たまたま月曜に動いた大型案件の応募が1件入っただけ、というケースが多い。回避策：施策判定前に「曜日別の過去3ヶ月平均」を散布図で可視化。単発の動きはノイズとして除外。
- **ダッシュボード過信：自動集計された数値をそのまま報告する危険性**：GA4やAirworkのダッシュボードウィジェットは「最新」「リアルタイム」をうたうが、キャッシュのタイムラグ、フィルター設定の誤り、ユーザー権限の不一致でしばしば不正確。月次レポート用のデータを取得する際、dashboard値をスクリーンショットそのままコピペすると、実データとの誤差が3-5%生じることもある。回避策：重要指標は必ず元データをエクスポート＆手作業で計算確認。ダッシュボード値と手計算値の乖離チェックを月初ルーチン化。

### 2026-05-07
- **Yui（SNSバズ分析）との「データ粒度」の事前統一**：Yuiが提供するSNS数値（インプレッション・リーチ・エンゲージメント）は日別・時間別の細粒度だが、GA4・Airworkデータは日単位・月単位で粗い。Yui のバズ検出（「この日のツイートが1時間で5000インプレ」）とGA4の応募数が対応しないことがある。回避策：Yui からデータをもらう前に「集計単位（日別 / 週別 / 月別）」「外れ値判定基準（過去平均の何倍以上をバズと見なすか）」「サンプル数最低閾値（1時間のインプレ数が100未満は除外）」をExcelで共有フォーマット化。一度決めたら全月同じ基準。

- **SNS単発情報 ≠ 市場トレンド決定の壁**：Yui からX・Instagramでバズっているキーワード（「建設業 AI導入」）をもらい、「これからAI関連の採用訴求が流行る」と先走り。実は特定アカウントの一時的な発言で、業界全体のトレンドではないケースが多い。Yui との引き継ぎ時に「このバズのトレンド強度」（フォロワー数・言及企業数・継続期間）を明記してもらうことで、施策判定の根拠が強化される。

### 2026-05-09
- **GA4「セッション」「イベント」定義の混同がクライアント説明で信頼を失う**：GA4 dashboard で「セッション数」と「ページビューイベント数」を混同して報告することがある。セッション = ユーザーが LP に訪問してから離脱までを 1 単位とする時間枠だが、複数ページを閲覧すれば複数イベント。「セッション 100 = 応募 5」と「イベント 100 = 応募 5」では分母の意味が全く異なり、CVR 計算に 20-30% の誤差が生じる。クライアント報告で「セッション」と言いながら「イベント」ベースで計算すると、クライアント側で検証時に乖離を発見し、信頼を失う。回避策：月初の「分析定義書」で「本月の CVR = セッション数 / 応募数」と明記。GA4 dashboard から数値を引く際に、必ず「このメトリクスの分子・分母は何か」を確認表に記録。
- **Airwork フロー定義「応募」「面接」「内定」の正式区分を統一しないと月次比較が破綻**：Airwork 管理画面では「応募完了」「一次面接進出」「二次面接」「内定」など段階があるが、クライアント企業ごとに「面接」の定義が異なることがある。ある企業は「書類選考 = 一次面接」と呼び、別の企業は「書類選考」と「面接」を分ける。Shun が「採用ファネル：応募 100 → 面接 30 → 内定 5」と報告したとき、クライアント側の「面接」定義と異なると「なぜ我社の面接進出数と違う？」という質問が返ってくる。回避策：Airwork データ抽出前に、クライアントのステータス管理を確認し「この企業では『面接』= Airwork のどのステータスか」を明文化。フロー図を 1 枚作成し、全レポートに付記。
- **CVR 計算の「分母」が業界・企業で異なる落し穴**：「応募 CVR = 応募数 / (セッション数・ユーザー数・ページビュー数)」の分母選定が企業によってまちまち。求人サイト経由では「ページビュー」を分母にするのが慣例だが、SNS経由では「セッション」を採用。複数チャネルを集計する月次レポートで、「今月の全体 CVR」を出す際に分母の定義が曖昧だと、Haruto や Ryota に説明する際に「何を根拠に出したのか」がぶれる。回避策：レポート作成時、チャネルごとに「CVR 計算の分母」を表で明記。「求人サイト = PV ベース、SNS = セッション ベース」と定義固定。月次レポートの「全体 CVR」集計時も、分母の選定理由を 1 行注釈。
